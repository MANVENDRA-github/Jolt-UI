# DECISIONS — Jolt UI (Architecture Decision Log)

Append-only. One entry per non-trivial choice so future sessions don't relitigate. Newest at the bottom.

---

### D-001 — Direction: multi-framework gallery (2026-06-27)

Ship every component for **React + Vue + Svelte** from one design language, rather than a React-only react-bits clone. **Why:** react-bits/Aceternity/Magic UI are React-only; their Vue/Svelte ports are separate, drifting repos. Unifying three frameworks from one source is the defensible, ownable angle. **Cost:** ~3× per-component work, mitigated by a shared core + scaffolder.

### D-002 — Site framework: Astro 5 (2026-06-27)

**Why:** Astro is the only mainstream meta-framework that renders React + Vue + Svelte islands on a single page (islands architecture) — exactly the live tri-framework demo need. Static output, great SEO/perf, MDX + content collections. Considered Next.js (React-only for the site, awkward for Vue/Svelte demos) and Vite SPA (weaker docs/SEO).

### D-003 — Monorepo: pnpm workspaces, source ("just-in-time") packages (2026-06-27)

Packages export raw TS/`.vue`/`.svelte` source (no build step); the site/tests compile them via Vite + framework plugins. Astro `vite.ssr.noExternal: ['@jolt/*']` so they're transformed server-side. **Why:** zero build ceremony for a solo dev; one source of truth. No Turborepo yet (task graph is small) — revisit if build caching hurts.

### D-004 — Delivery: jsrepo registry + copy-paste (2026-06-27)

**Why:** jsrepo is framework-agnostic (multi-framework blocks), supports copy-paste + `npx jsrepo add`, and is what react-bits uses. shadcn CLI is React-centric. Implemented in Phase 1+.

### D-005 — v1 variants: TS + Tailwind only; Tailwind v4 CSS-first (2026-06-27)

Cut react-bits' 4-variant matrix (JS/TS × CSS/TW) → 12 files/component/3 frameworks. v1 ships **TS + Tailwind** per framework. **Why:** simplicity-first; JS/plain-CSS variants are mechanical and scaffold-generatable later. Tailwind **v4** chosen specifically: CSS-first `@theme` lets one `tokens/theme.css` be shared across all packages with no JS preset duplication.

### D-006 — Hand-written type shims for cross-package Vue/Svelte (2026-06-27)

`packages/{vue,svelte}/types.d.ts` declare the public component types by hand; `exports.types` points at them so `astro check` can type cross-package `.vue`/`.svelte` imports (Astro's checker has no Vue/Svelte language tooling). The **Svelte** shim is typed **props-first** (`(props) => unknown`) because Astro reads a component's first parameter as its props, whereas Svelte 5's `Component<P>` puts internals first. **Replace with generated `.d.ts` when a build step is added.**

### D-007 — Pin Vite to ^6; vitest's internal Vite 7 tolerated (2026-06-27)

Astro 5 uses Vite 6; `auto-install-peers` was pulling Vite 7 for plugin peers, causing a type clash in `astro.config.ts` (Tailwind plugin typed against Vite 7 vs Astro's Vite 6). Fixed by `pnpm.overrides.vite=^6` + an explicit root `vite@^6` devDep + reverting the Vue/Svelte Vitest plugins to their Vite-6 lines. **Residual:** vitest's `@vitest/mocker` still resolves Vite 7 (an auto-installed peer pnpm overrides don't constrain). Harmless — isolated to the test runner; site build, `astro check`, and tests are all green. Revisit if it ever bites.

### D-008 — Animation: CSS-first, GSAP when needed (2026-06-27)

GSAP became 100% free (incl. SplitText with built-in a11y, ScrollTrigger) in 2025 and is framework-agnostic → ideal shared core. Used as a **peer dependency**, registered client-side, only on components that need it. Three.js deferred to the Backgrounds category. Implemented in Phase 1.

### D-009 — Node ≥ 20.3 (2026-06-27)

Engines target Node ≥ 20.3 (the dev machine runs 20.20.1; satisfies Astro/Tailwind v4/Vitest). CI runs Node 20. Bump to 22 LTS later if a dependency requires it.

### D-010 — Name "Jolt UI", scope `@jolt/*` (2026-06-27)

Project named **Jolt UI** (was the placeholder "Triad"). npm/registry scope `@jolt/*`; CSS token prefix `--jolt-*`; CLI intent `npx jolt add` (Phase 1+). Canonical home is the GitHub repo `MANVENDRA-github/Jolt-UI`, cloned at `D:\Jolt-UI`; the throwaway scaffold dir `D:\triad` was migrated here and removed.

### D-011 — Vue skin declares props locally (2026-06-28)

Vue's `@vue/compiler-sfc` cannot resolve a schema-inferred type (`z.input<typeof splitTextSchema>`) inside `defineProps<T>()` — it throws "Unresolvable type reference". React and Svelte compile the inferred type fine. **Resolution:** the Vue SFC declares a local `Props` interface that mirrors the schema (kept in sync by hand + comment). The Zod schema remains the single source for runtime validation/defaults and for React/Svelte prop types; only Vue duplicates the _shape_. A future guard: a Vue-package test asserting the local interface keys match `splitTextSchema.shape`.

### D-012 — Registry: bundle the core ("own 100%") via a build transform (2026-06-28)

Distribute components via **jsrepo** so consumers own all the code, including the shared core (no npm runtime dep). Key jsrepo (v3.8) behaviors discovered empirically:

- jsrepo treats a package-like specifier (`@jolt/core`) as an **npm dependency** and will not bundle it. Only **path aliases** (e.g. `@/jolt-core`) are resolved to local files and bundled (with the import rewritten on `add`).
- jsrepo resolves path aliases from a `tsconfig.json` at the **config cwd (repo root)** — per-package tsconfig `paths` were ignored. So a root `tsconfig.json` declares `@/jolt-core` → `packages/core/src/index.ts`.
- A jsrepo **`build.transforms`** entry runs **before** import resolution, so we rewrite `@jolt/core` → `@/jolt-core` at build time. The **source keeps `@jolt/core`** (monorepo runtime, Vitest, and type-checking are unchanged); only the distributed blocks use the alias.

**Shape:** one registry per framework (item names must be globally unique) → `apps/site/public/r/{react,vue,svelte}`. Each has a `split-text` component item + a shared `core` lib item; `split-text` declares `core` as a registry dependency. Core files use the glob `packages/core/src/**/!(*.test).ts` — subdirs preserved, **tests excluded** (jsrepo has no file-ignore; `target` and negative-glob entries did not work; the extglob did).

**Plumbing:** the output is git-ignored and regenerated by `registry:build` (run as part of `build`) and validated by `registry:check` (part of `verify`; asserts core is bundled, no `@jolt/core` npm dep, import rewritten, no test files leaked).

### D-013 — CSS-only components: one shared stylesheet in the core (2026-06-28)

CSS-only animations (Blur In, Wave, …) keep their `@keyframes` + classes in a **shared module** `packages/core/src/styles/<id>.css` — the single source so all three skins render identically (no per-framework CSS drift). The skins set tunable values as `--jolt-*` **custom properties** from props and `import '@jolt/core/styles/<id>.css'`. **Why one source:** the differentiator is cross-framework parity; duplicating CSS per skin would reintroduce the drift we exist to prevent.

- **Resolution/typecheck:** added a `@jolt/core/*` subpath export (`"./*": "./src/*"`) + a root `@/jolt-core/*` alias; each framework package has a `src/css.d.ts` (`declare module '*.css'`) so `tsc`/`vue-tsc`/`svelte-check` accept the side-effect import.
- **Registry bundling:** the CSS files are part of the **`core`** item (added a `styles/*.css` glob); the skin's CSS import resolves to a registry file, so `jsrepo add <id>` bundles core + the stylesheet and rewrites the import to a local path on `add`. Validated end-to-end by CLI-smoke (adds the component, asserts the `.css` lands, consumer type-checks). **Trade-off:** the core item is monolithic, so any component install pulls the whole core (all schemas + every CSS). Acceptable while core is small; revisit with per-component slices if it grows.
- Generalized alongside this: `propsTable(schema)` (was `splitTextPropsTable`), `registry-check.mjs` loops **all** component items, and the parity harness/spec is now a combined `/internal/parity` + `e2e/parity.spec.ts` that loops over component ids.

### D-014 — Release `will-change` in the reduced-motion final state (2026-06-28)

A CSS-only segment animating `filter`/`transform` carries `will-change` to hint GPU compositing. The parity E2E (which runs under `prefers-reduced-motion: reduce`) failed for Blur In at ~2.2% because the static segments were **still GPU-composited** (`will-change` left on), and per-glyph layer rasterization differs subtly between the React and Vue cells. **Fix:** the `@media (prefers-reduced-motion: reduce)` block sets `will-change: auto` (alongside `animation: none`), so the static text rasterizes on the deterministic CPU path. SplitText (GSAP) didn't hit this because GSAP clears the hint at its final state. **Rule:** any CSS-only component must neutralize `will-change` in its reduced-motion state, not just the animation. (This makes the _static_ frame deterministic; **looping** animations need D-015 on top.)

### D-015 — Parity harness freezes animations to a deterministic frame (2026-06-28)

D-014 made each component's static frame render deterministically, but the parity E2E still failed **on CI only** for Wave at ~2.2% (green locally). Root cause, confirmed by a diagnostic: finite animations (GSAP SplitText, the one-shot Blur In) settle to a static frame before capture, but Wave's **infinite** CSS bob only freezes if Playwright's reduced-motion emulation is honored at paint time — reliable on local Windows Chromium, **not** on headless CI Linux Chromium, where the still-running loop was screenshotted at slightly different phases per cell. (The static layout itself is provably identical: relative segment rects match exactly and the frozen pixel diff is 0.) **Fix:** before comparing, `e2e/parity.spec.ts` injects a stylesheet collapsing every animation to its end-state (`animation-duration: 0s; animation-iteration-count: 1; animation-fill-mode: forwards; transition-duration: 0s`), so all three frameworks are captured at the same frozen frame regardless of environment. Inert for the finite/GSAP components. **The components keep their real looping animations — only the test freezes them.** **Rule:** parity-test looping components by freezing in the harness; never rely on reduced-motion emulation alone for a deterministic frame.

### D-016 — Whole-text CSS-only components (no segmentation) (2026-06-28)

Some CSS-only effects animate the **whole string** rather than per-character — Gradient Text and Shiny Text (animated `background` + `background-clip: text`). These skip segmentation entirely: the skin renders `<span class="jolt-…">{text}</span>` directly, so the text is **natively accessible** — no `aria-label` / `aria-hidden` segments (those exist only to keep per-char split text readable). Distribution is identical to D-013 (shared `styles/<id>.css`, `--jolt-*` from props, bundled via the core item). Consequences:

- **The parity harness/spec now handles two component kinds:** _per-char_ (assert aria-label + segment-count parity) vs _whole-text_ (assert text-content parity only). Read the accessible text from the text-bearing element — the aria-label span for per-char, the component's own span for whole-text — **never the whole cell**, because a `client:load` island (e.g. SplitText) embeds an inline hydration script whose text would otherwise leak into `textContent` and break the comparison.
- **`propsTable` gained `ZodArray` support** (renders `string[]`) for Gradient's `colors` prop.
- Reduced-motion still renders a sensible static state, and D-015's harness freeze still applies to the looping background animation.

### D-017 — Structural CSS-only components (clip + `steps()` width/translate reveal) (2026-06-28)

A third CSS-only sub-pattern beyond per-char (D-013/D-014) and whole-text (D-016): **structural** components whose effect comes from DOM structure + an `overflow` clip rather than per-glyph keyframes. Two shipped in PR 2c:

- **Typewriter** (whole-text parity kind): an `overflow:hidden; white-space:nowrap` box animates `width: 0 → calc(var(--jolt-steps) * 1ch)` in `steps(var(--jolt-steps))`, revealing one character per step; a `border-right` caret blinks via a separate `step-end` animation. Requires a **monospace** font (so `1ch` == one advance and the steps land on character boundaries) and `box-sizing: content-box` (so the caret border isn't subtracted from the content width under a global `border-box` reset, which would clip the last char). The skin sets `--jolt-steps = text.length`. Reduced-motion and the harness freeze both resolve to full static text with a transparent caret.
- **Rotating Words** (per-char parity kind): a vertical column of word spans inside a one-line clip; the list translates up one line per interval in `steps(var(--jolt-count))` — a **discrete flip**, chosen over a continuous slide so each word is held fully readable. A **trailing duplicate of the first word** makes the loop wrap seamlessly and gives the frozen parity end-state a real word to show (cf. D-016's repeated gradient stop). Takes `words: string[]`; the `aria-label` is the **space-joined** words so the parity spec's per-char locator (`[aria-label="Jolt UI"]`) matches when the harness passes `words={['Jolt','UI']}` — no spec-logic change, just `'rotating-words'` added to `PER_CHAR`. The animated list is `aria-hidden`; each real word carries `data-jolt-segment` (the duplicate does not, so segment count == `words.length`).

**`steps(var(--…))`:** modern Chromium resolves a CSS custom property inside `steps()` (validated live this session: `steps(3)` for 3 words, `steps(21)` for a 21-char string). It governs only the _live_ look — the parity harness (D-015) freezes animations to their end-state — so it carries no gate risk; were a future target engine to reject it, the skin would set `animation-timing-function` inline from the same count (parity-safe). Distribution is identical to D-013/D-016 (shared `styles/<id>.css`, `--jolt-*` from props, bundled via the core item).

### D-018 — GSAP CountUp + Scramble; whole-text GSAP parity + harness robustness (2026-06-28)

PR 2d added the second GSAP slice on the `createSplitText` pattern (factory → `{ play, revert }`; skins create on mount + `revert` on unmount; `prefersReducedMotion()` jumps to the final state).

- **CountUp** tweens a plain `{ value }` object with `gsap.to`, writing `formatNumber(value, { decimals, separator })` into the element on each `onUpdate`; reduced-motion / `revert` show the formatted target. `formatNumber` is a **pure core export** reused by the skins for their SSR initial value, so there's no per-skin formatting drift. Tweening a JS object (not the DOM) lets the core test exercise the real animated path in jsdom.
- **Scramble** uses **ScrambleTextPlugin** — `import { ScrambleTextPlugin } from 'gsap/ScrambleTextPlugin'` + `gsap.registerPlugin(...)` at module load **in `animation/scramble.ts`** (not `motion.ts` as first sketched in the roadmap) so the plugin loads only with Scramble, leaving SplitText's path + green tests untouched. Verified SSR-safe (the dev server + parity harness SSR the module without error) and that the `gsap/ScrambleTextPlugin` subpath resolves in a real consumer (`pnpm test:cli` typecheck — jsrepo auto-installs `gsap`). `aria-label={text}` keeps the real text accessible while the visual churns.
- **Whole-text GSAP parity (new):** both are the first **whole-text** components rendered via `client:load`. A GSAP component is deterministic for parity **without** the D-015 CSS freeze, because it reads `prefersReducedMotion()` (`matchMedia().matches`) — which Playwright's `reducedMotion: 'reduce'` honors in JS reliably on local + CI — so the factory jumps to the final value/text. CountUp's harness cell passes `to={100}` (it has no `text` prop); Scramble uses the shared `text={TEXT}`.
- **Parity-harness robustness (test-only; components render identically):** GSAP animations aren't frozen by the D-015 CSS stylesheet (they're JS), so on **headless CI Linux** (local Windows passed) SplitText could be captured mid-animation or with per-glyph GPU-layer antialiasing differences. Three harness changes make it deterministic: (a) the freeze stylesheet adds `will-change: auto !important` so per-char segments CPU-rasterize identically across frameworks (D-014, generalized to the harness); (b) before comparing, the spec **waits for the longest GSAP island (CountUp, 2s) to reach its target** — deterministic whether or not reduced-motion is honored in JS (honored → instant; not → the real tween finishes), by which point SplitText (~0.8s) and Scramble (1.5s) are settled too; (c) Astro's `position: fixed` **dev toolbar** is **disabled** during E2E (`devToolbar.enabled` gated on a `JOLT_E2E` env that Playwright's `webServer` sets; left on for a normal `pnpm dev`) — it bled into element screenshots as the page grew taller. A CSS `display:none` hide proved unreliable on CI, so it's disabled at the config level instead. Diagnosed by screenshotting the cells + reading the pixelmatch diff (which highlighted the toolbar, not the component text).

### D-019 — ScrollVelocity (GSAP ScrollTrigger) + parity for an infinite marquee (2026-06-28)

PR 2e added the last Text-Animation component and the `/components` index page (Phase 2 complete).

- **ScrollVelocity** is a horizontal marquee: the skin renders the text in an `aria-hidden` track of repeated copies (`aria-label` carries it once), and the factory animates the track `xPercent: 0 → -50` for a seamless loop (the copies are symmetric, so the second half takes the first's place). **ScrollTrigger** reads `self.getVelocity()` on scroll to boost `timeScale` and flip direction, easing back to the idle speed when scrolling stops.
- **ScrollTrigger registration is lazy + client-only.** Unlike ScrambleTextPlugin (D-018), `gsap.registerPlugin(ScrollTrigger)` calls `window.matchMedia` — absent in jsdom (tests) and on the server (SSR) — so registering at module load threw `_win.matchMedia is not a function`. Registration runs **inside the factory, past the `prefersReducedMotion()` early-return**, so it only fires in a real browser when actually animating. The `gsap/ScrollTrigger` import itself is SSR-safe (GSAP guards `window` at eval).
- **Parity for an infinite, repeated-text marquee.** Two problems: (1) a looping GSAP animation isn't frozen by the D-015 CSS stylesheet (that only touches CSS animations) and never settles, so the count-up settle-wait (D-018) doesn't help; (2) more fundamentally, the marquee **repeats** its text, so a sub-pixel cross-platform layout difference accumulated over the copies misaligns the repeating pattern and explodes the pixel diff — measured locally a 1px shift ≈ 2.4% (already over the 2% threshold) and 8px ≈ 24% (the CI diff, i.e. an ~8px phase offset). Pinning the transform can't fix subpixel _width_ accumulation, so ScrollVelocity is **excluded from the pixel comparison** (`NO_PIXEL_PARITY` in `e2e/parity.spec.ts`). Its anti-drift comes from the **DOM/text-parity** check (identical text content across frameworks) + the per-framework **unit tests** (identical markup: `aria-label` + an aria-hidden track of N copies). Classified **whole-text** (no segments).
- **Pixel-parity threshold raised 1% → 2%.** Subpixel antialiasing between independent React/Vue/Svelte renders of the _same_ per-char layout hovers ~1% even on one machine (Wave flaked across the 1% line on a re-run). Real cross-framework drift (missing element, wrong text/layout) is far larger, so 2% is stable without weakening the gate.
- **`/components` index** (`apps/site/src/pages/components/index.astro`): a card grid with a live React mini-preview per component, linking to each demo. The homepage now links to it; the Phase-0 Hello smoke test is retired (the `Hello` component + its unit tests stay).

### D-020 — Install UX derived from `meta` (single source) + `zod` is a universal peer (2026-06-28)

Phase 3 PR 1. Each component demo page **hardcoded** its install block — GSAP pages a `pnpm add gsap zod` tab, the CSS-only pages no peer-deps line at all. That drifts, and for CSS-only it under-documents: the bundled **monolithic** core's schemas all `import { z } from 'zod'` (D-013), so `zod` is a runtime peer for **every** component, not just the GSAP ones. **Resolution:** a pure `installInfo(meta, framework)` helper in `@jolt/core` (sits beside `propsTable` — a docs helper that already ships in the bundle) derives the install commands + dependency list from the component's `meta`: `peers = dedupe([...meta.deps, 'zod'])` (zod always; `gsap` iff the meta declares it), `registryPath = ${REGISTRY_BASE}/r/${framework}/${id}`, and one npm + one pnpm tab (`jsrepo add` line + the peer-deps install). A single `InstallBlock.astro` renders those tabs (via the existing `CodeTabs`) plus the dependency badges, so every page is now just `<InstallBlock meta={<id>Meta} />` — the install UX can't drift from what `jsrepo add` actually pulls.

- **`REGISTRY_BASE`** centralizes the registry origin as the placeholder `<registry-url>`; the Phase 4 deploy swaps that one value.
- **Anti-drift test:** a core unit test loops every exported `*Meta` and asserts `peers` always includes `zod` and includes `gsap` iff `deps` does (mirrors the `propsTable` no-drift test). An `e2e/install.spec.ts` asserts the rendered badges + `jsrepo add` command for a GSAP page (gsap + zod) vs a CSS-only page (zod only).
- **Consequence (intended fix):** CSS-only pages now correctly surface `zod` as a peer dependency (previously omitted entirely).

### D-021 — `gen-component` scaffolder: `.mjs`, contract-driven, working-green stub (CSS) (2026-06-28)

Phase 3 PR 2. Adding a component meant creating ~13 files and hand-editing ~8 central registration files (the `COMPONENT_GUIDE.md` checklist). `node scripts/gen-component.mjs <id>` now stamps the whole slice from one declarative contract (`scripts/contracts/<id>.mjs`), validated at runtime by `assertContract`.

- **Language `.mjs`, not the roadmap's `gen-component.ts`:** Node 20.20 can't run TS and there's no tsx/ts-node; every script is `.mjs`. Type-safety comes from the runtime-validated contract **plus** the generated output being gated by `pnpm verify` (tsc/vue-tsc/svelte-check/astro check) + `test:cli` + `test:e2e`.
- **Functional core / imperative shell:** pure `scripts/gen/{contract,emit,edits}.mjs` (validator; `contract → string` emitters; idempotent marker splices), unit-tested by a new **`test:gen`** (`node --test`, 45 tests) wired into `verify`; the thin shell does fs + a `prettier --write` pass.
- **Anchor-marker edits + all-or-nothing precheck:** the 8 central files carry inert `// gen:*` / `{/* gen:* */}` markers; `insertBeforeMarker` asserts a marker occurs exactly once (fail-visible if hand-removed). Before any write the shell scans every target for the id's sentinel and aborts with **zero writes** if it's already generated (re-runs never duplicate or partially write).
- **Working-green stub:** the generated component passes `verify` + `test:cli` + `test:e2e` **immediately** — the structural invariants (aria-label, segment count, `--jolt-*` mapping, cross-framework parity) are identical across all components, so they ship as passing tests; the animation is a trivial opacity fade (incl. the D-014 reduced-motion `will-change: auto`) the author then customizes **test-first**. Proven end-to-end this session: generated a throwaway `demo-fade`, ran all three gates green, reverted.
- **Scope:** CSS patterns only (`css-per-char` / `css-whole-text` / `css-structural`; the skin markup branches on `parity.kind`). `gsap` is rejected with a hand-write hint — GSAP factory + lifecycle scaffolding is a fast follow. The contract requires a `text` string prop (per-char also a `by` enum); bespoke components (count-up's number, rotating-words' word list) stay hand-written.

### D-022 — Deploy: Cloudflare Pages (static, Git integration); single `JOLT_ORIGIN` (2026-06-28)

Phase 4 PR 1 (origin) → PR 5 (deploy config). The site is a fully **static** Astro build and the jsrepo registry builds into `dist/r/*`, so a static **Cloudflare Pages** deploy serves the site **+ `/r/*` from one origin** — no Worker/SSR adapter. Deploy via CF's **Git integration** (connect the repo; CF runs `pnpm build` → `apps/site/dist` on push, free per-PR previews) — chosen over a GitHub-Actions + `wrangler` job (no secrets, no CI surface a solo dev needs). The deployed origin must be both the registry base **and** astro's `site` (canonical/OG/sitemap), so one `packages/core/src/origin.ts` exports `JOLT_ORIGIN` (provisional `https://jolt-ui.pages.dev`); `REGISTRY_BASE = JOLT_ORIGIN` and `astro.config` `site: JOLT_ORIGIN` (imported by a **relative** path — the `@jolt/core/origin` subpath export doesn't resolve in the config's TS context). One value swaps at deploy (PR 5 adds `public/_headers` + `DEPLOY.md`).

### D-023 — SEO: pure `seo.ts` head helpers + `@astrojs/sitemap` (2026-06-28)

### D-035 — ProgressBar opts out of pixel parity (no stable frozen frame) (2026-06-29)

Phase 7 PR 7c follow-up (fix). The indeterminate **ProgressBar** loader is the first `GRAPHIC` component whose pixel parity is **unreliable**, so it joins `NO_PIXEL_PARITY` in `e2e/parity.spec.ts` — the second member, after scroll-velocity (D-019). **Why:** the harness pixel-compares each framework's cell at a frozen frame, but ProgressBar has **two very different static states** — its reduced-motion state is a 45%-width fill at the left; its animation end-state (the freeze target, D-015) sweeps the fill off the track, leaving the bare rail. Playwright's context-level `reducedMotion: 'reduce'` is **not applied atomically across the three sequential per-cell screenshots** (it's flaky in general — the 7b live-verify switched to `page.emulateMedia`), so one cell can be captured showing the 45% fill while another shows the bare rail → an ~12% diff that trips the 3% threshold **intermittently** (passed on #34's CI, failed on #35's). This is **not drift**: the rendered DOM is byte-identical across React/Vue/Svelte and the stable diff is **0%** (verified by dumping all three cells' `innerHTML` + screenshots). Anti-drift for ProgressBar is therefore the **shared CSS** (one stylesheet, three skins) + the **per-framework unit tests** (role=status, one fill span, `--jolt-*` mapping) + the `GRAPHIC` **visibility** assert — exactly the scroll-velocity rationale (D-019). The other six loaders keep pixel parity (their reduced-motion and freeze states stay close enough to hold under the threshold). **General rule:** a component whose reduced-motion static state diverges sharply from its animation end-state cannot be freeze-pixel-compared and belongs in `NO_PIXEL_PARITY`. (Process note: CI runs `test:e2e`, not just `verify`, and the repo has no branch protection — #35 merged over a red check; future merges gate on the actual CI conclusion.)

### D-036 — Buttons: the 4th category + the `INTERACTIVE` parity kind + the interactive skin pattern (2026-06-29)

Phase 8. Buttons (Shimmer, Glow, Gradient, Sweep, Border Draw, Tactile) are the gallery's fourth category and its **first interactive components** — a real `<button>` with a text label, animated by interaction (hover/press) or a self-running surface effect. Three new things:

**(1) The interactive skin pattern.** The `@jolt/core` schema carries **only** the visual `--jolt-*` contract + a `label` fallback (the one design language); it does **not** model `disabled`/`onClick`/`type`/`children` — those are native `<button>` concerns each skin forwards **idiomatically**, so a consumer gets a real, fully-featured button. **React:** `Props = <id>Props & ButtonHTMLAttributes<HTMLButtonElement>`; destructure the visual props + `type`/`className`/`style`/`children` out, spread `...rest` onto `<button>`, merge `--jolt-*` into `style`, render `{children ?? label}`, default `type='button'` (no `any`; `children` comes free from `ButtonHTMLAttributes`). **Vue:** a single root `<button><slot>{{ label }}</slot></button>` with the local `Props` interface (D-011) declaring only the visual props + `label`; native attrs (`@click`/`disabled`/`type`/`aria-*`) reach the button by **attribute fallthrough** (`inheritAttrs` default true) — **no `defineEmits`** (it would make `onClick` a declared emit and break fallthrough). **Svelte 5:** `Props = <id>Props & Omit<HTMLButtonAttributes,'children'|'class'|'style'> & { children?: Snippet; class?: string; style?: string }`; destructure visual + `type`/`class`/`style`/`children` out (so `...rest` can't clobber the base class) and `{#if children}{@render children()}{:else}{label}{/if}`. The hand-written `types.d.ts` shims add only `disabled?: boolean` to the visual props (Astro SSR previews attach no handlers and pass children regardless of the props type).

**(2) The `INTERACTIVE` parity kind** in `e2e/parity.spec.ts`: assert every framework cell renders a `<button>` + the label text matches across frameworks, then **pixel-compare the REST state**. The existing freeze (D-015) collapses a self-running effect (shimmer/glow/gradient/border-draw) to a deterministic frame; a hover/press effect (sweep/tactile) renders as its static base at rest, since the spec **never triggers** `.hover()`/`.focus()`/`.click()`. Reusing `GRAPHIC` was rejected (it skips the text assert, and the label — `children ?? label` vs `<slot>` vs snippet — is the button-specific drift risk); `WHOLE_TEXT` doesn't fit (it reads `span.first()`, but a button label is a direct child of `<button>`). **Interaction-state parity (hover/focus/press screenshots) is explicitly deferred**; the behavior contract (click fires, click forwards, `disabled` reflected, keyboard) is covered by the per-framework **unit tests** (the first to use `fireEvent`) + the shared-CSS single source. The "does not fire when disabled" assertion was dropped: jsdom dispatches `fireEvent.click` on a disabled button to native listeners (Vue/Svelte) even though a real browser suppresses it, so it tested jsdom not the component — the meaningful contract is that `disabled` is reflected on the native `<button>` (the browser does the suppression). Vue unit tests pass fallthrough props (`onClick`/`disabled`) via the render `attrs` option, not `props` (vue-tsc types `props` against the declared props only).

**(3) The index card.** A live `<button>` must **not** be nested inside the card's `<a>` (invalid HTML + a double tab stop), so the Buttons `<section>` in `/components` uses a `<div>` card with the live button preview + a **sibling** “View →” link — unlike the fully-clickable `<a>` cards of the other categories.

Buttons are **hand-written** (the `gen-component.mjs` scaffolder stays text-only, per D-034); category bootstrap is one `'button'` entry in `apps/site/src/lib/categories.ts` (slug `buttons`). Names avoid the loader ids (`pulse`/`ripple`/`grid` are loaders); `gradient` is distinct from the text `gradient-text`. **Pixel-parity split (confirmed by CI):** the four **self-running keyframe** buttons (`shimmer`/`glow`/`gradient`/`border-draw`) join `NO_PIXEL_PARITY` — they share ProgressBar's flaw (D-035): their reduced-motion static frame (gradient at 0% / a steady glow) diverges sharply from the freeze's keyframe end-state (gradient swept to ±200% / a small glow), and Playwright's context-level `reducedMotion` isn't applied atomically across the three per-cell screenshots, so on headless-Linux CI cells get captured in different states → a large, intermittent full-surface diff (shimmer react-vs-svelte tripped the 3% threshold on the 2nd CI run while the 1st passed — the intermittent signature). The two **transition-only** buttons (`sweep`/`tactile`) keep pixel parity — no `@keyframes`, so their rest frame is identical with or without reduced-motion. Anti-drift for the excluded four = shared CSS + per-framework unit tests + the `INTERACTIVE` `<button>`/label-text assert. Live-verified in a real browser (17 checks): the 4 self-running buttons animate + reduced-motion stops the glow; sweep changes color on hover; tactile's press transition is wired; the button is keyboard-focusable and a native click fires; no console errors.

### D-040 — Scaffolder v2: the `container` (card) + `interactive` (button) kinds (2026-07-10)

Phase 10a. `scripts/gen-component.mjs` could only stamp **text** components, so every loader (D-034), button (D-036), and card (D-037) since Phase 7 was hand-written — three categories' worth of the same 12-file slice, typed out by hand. Phase 10 adds cards, buttons, and two new container-shaped categories, so the scaffolder is extended to the two kinds whose skins are now _proven and stable_: **container** (a presentational `<div>` wrapping children — Spotlight/Tilt/ShineBorder) and **interactive** (a native `<button>` with a `label` fallback — Shimmer/Sweep/Tactile). **Loaders (`GRAPHIC`) and Three.js backgrounds stay hand-written** — a loader's markup is bespoke per effect (N decorative children), and a background is a WebGL factory, not a stylesheet. **GSAP stays rejected** (still the long-standing follow-up).

**(1) The contract's text-only rules become kind-conditional.** `contract.mjs` previously hard-required a `text` string prop and ≥1 required prop. Both are text-kind invariants: `spotlightSchema` and `shimmerSchema` are entirely defaulted (zero required props), a container carries neither `text` nor `label`, and an interactive carries `label` (children/slot is the real content, `label` the fallback — D-036). So: `PATTERNS` gains `css-container`/`css-interactive`, `PARITY_KINDS` gains `container`/`interactive`, the `text`/required-prop rules are gated on `isTextKind(parity.kind)`, an interactive must declare a `label` string prop, and `label` joins `text`/`by` in the set exempt from the "every CSS prop needs a `cssVar`" rule. Pattern and parity kind are cross-checked in both directions — a `css-container` skin behind an `INTERACTIVE` parity assert would emit a `<div>` and then look for a `<button>`.

**(2) Two new contract fields: `category` and `hydrate`.** `emitSchema` hardcoded `category: 'text'`; the contract now carries it (`text | card | button | effect | ui`, defaulting to `text` so the three existing text contracts are untouched), and `categorySlug()` maps it to the plural URL segment (`cards`, `buttons`, `effects`, `ui`) used by the demo-page path, the index-card `href`, and the `gen:card:<slug>` marker. `hydrate: true` marks a **cursor-tracking** container: its parity cell gets `client:load` and its previews `client:visible`, because the pointer behavior only writes the deterministic rest state once hydrated. The scaffold itself emits **no pointer wiring** — the writer (`writeSpotlight`, `makeTiltWriter`, …) is component-specific, so `trackPointer` is added by hand in the same test-first customization step as the keyframes.

**(3) Emitters clone the shipped skins, not a new shape.** The container/interactive branches of `emitReactSkin`/`emitVueSkin`/`emitSvelteSkin` reproduce Spotlight/Shimmer exactly: React `& HTMLAttributes<HTMLDivElement>` / `& ButtonHTMLAttributes<HTMLButtonElement>` + `...rest` (no `any`); Vue single-root + **attribute fallthrough**, no `defineEmits`; Svelte 5 `& Omit<…, 'children'|'class'|'style'>` + a `Snippet`. One subtlety the emitter encodes: a generated Svelte `<button>` closes **tight** around its label (`>{#if children}…{/if}</button\n>`), because whitespace inside a `<button>` renders as a space and would widen the Svelte cell against React's in the parity pixel diff; a block `<div>` collapses its whitespace and can breathe. The interactive shims add `disabled?: boolean` (D-036); the generated Vue test passes fallthrough attrs via the render **`attrs`** option, Svelte via `props`.

### D-041 — Per-section gallery-card markers (`gen:card:<slug>`) (2026-07-10)

`apps/site/src/pages/components/index.astro` is hand-written per category, and `edits.mjs:insertBeforeMarker` requires its anchor to occur **exactly once** (a duplicate throws — the latent bug found and "fixed" in D-038(5) by deleting the second `{/* gen:card */}`). The consequence went unnoticed because the scaffolder was text-only: the single surviving marker sits in the Text section, so **no card could ever be spliced into any other category**. Scaffolding cards and buttons therefore requires one marker per section — `{/* gen:card:text */}`, `{/* gen:card:cards */}`, `{/* gen:card:buttons */}` (and one per future category) — with `applyComponentsIndex` selecting `gen:card:${categorySlug(c.category)}`. This **supersedes D-038(5)'s one-marker rule**: the invariant was never "exactly one marker in the file", it was "exactly one marker per insertion point". `containsComponent`'s idempotency probe (`/components/text/<id>`) is likewise now slug-aware — left unfixed, a re-run for a card would miss its own card and double-splice it.
